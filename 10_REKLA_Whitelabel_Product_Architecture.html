3. Sequence Diagram

VERY IMPORTANT

End-to-end local ride sequence with assignment, live tracking, OTP checks, and exception handling. GPS updates are streamed via SignalR (not direct to Redis) for security and access control.

Security gateway: Driver App connects to SignalR Service (authenticated WebSocket), NOT directly to Redis. SignalR mediates all access to streaming channels.
Event publishing pattern: All critical events (GPS updates, DriverArrived, TripStarted, TripCompleted, ForceEnd) are published to Redis pub/sub channels. SignalR subscribes to these channels and fans out to connected Rider and Admin clients in real-time.
Business logic at Core Service: Arrival, trip start/end, OTP verification, and fare computation remain in Core Service for audit and business rule enforcement. Core Service then publishes the outcome events to Redis.
Admin real-time visibility: Admin Portal receives all trip state events (arrival, start, completion, force-end) via Redis fan-out, enabling live monitoring without polling the core service.
Throttling: Driver App batches or throttles GPS sends (typically ~5s intervals). Other events (OTP, trip state) are sent immediately.
Control: Single active ride per driver enforced during accepted and in-progress states. Trip state transitions validated by Core Service before event publication.

7. SOS / Emergency Alert Flow

SAFETY CRITICAL

Rider can trigger SOS button during active trip to request immediate emergency assistance. Location and trip details are broadcast to emergency contacts and ops team for intervention.

sequenceDiagram autonumber participant R as Rider App participant C as Core Service participant RD as Redis (pub/sub) participant S as SignalR Service participant A as Admin Portal participant E as Event Bus participant EC as Emergency Contact R->>C: SOS Button Triggered (trip_id, location) C->>C: Log SOS event with timestamp and proof C->>RD: Publish SOSTriggered event (live location, trip details) RD->>S: Fan-out SOS to subscribed admins S-->>A: Real-time SOS alert + trip details + live map C->>E: Emit SOSAlert notification E->>EC: Send Emergency Contact SMS/Call with rider location link E->>A: Send Admin Dashboard push notification alt Admin Action - Contact Rider A->>C: Admin initiates call/chat with rider C->>S: Open support channel S-->>R: Incoming admin contact / chat request else Admin Action - Cancel Trip/Force End A->>C: Admin cancels or force-ends the trip C->>RD: Publish TripCancelled/ForceEnded via SOS RD->>S: Fan-out to rider and driver S-->>R: Trip cancelled due to SOS end
Real-time visibility: SOS events are immediately published to Redis and pushed to Admin Portal for instant visibility. No polling required.
External notification: Emergency contacts receive SMS/call with a link to rider's live location on a public map view (subject to privacy policy).
Audit trail: All SOS events, admin actions, and emergency contact notifications are logged for compliance and post-incident review.
Single SOS per trip: Once SOS is triggered, it locks the trip to prevent rider accidental re-triggers until trip ends or admin closes SOS.

8. Decline / Retry & Rider No-Show Handling

Driver decline workflow with reassignment logic and rider no-show detection when driver arrives but rider is absent.

sequenceDiagram autonumber participant D as Dispatch Engine participant DR as Driver App participant C as Core Service participant RD as Redis (pub/sub) participant S as SignalR Service participant R as Rider App participant A as Admin Portal D->>DR: Ride Request (60s accept window) alt Driver Accepts DR-->>D: Accept D->>C: Assignment confirmed C->>RD: Publish DriverAssigned event RD->>S: Fan-out to rider/admin S-->>R: Driver assigned with ETA else Driver Declines DR-->>D: Decline D->>C: Log decline event, increment driver decline count alt Decline Count < 3 D->>D: Select next eligible driver from pool D->>DR: Send ride request to next driver (loop back) else Decline Count >= 3 C->>RD: Publish DeclineThresholdReached event RD->>S: Fan-out to rider/admin S-->>R: Extended matching ETA S-->>A: Operational alert on high declines end alt Decline Count > 5 D->>D: Temporarily move driver to Offline C->>RD: Publish DriverOfflined event (too many declines) RD->>S: Fan-out to driver/admin S-->>DR: Driver moved offline due to repeated declines S-->>A: Driver compliance review alert end end DR->>C: Arrived at Pickup Location C->>RD: Publish DriverArrived event RD->>S: Fan-out to rider/admin S-->>R: Driver Arrived notification + 5 min countdown S-->>A: Driver arrived event alt Rider shows up DR-->>C: Rider confirmed / Scan/OTP for Start Trip note over C: Trip continues normally else Rider No-Show (>5 min wait threshold) DR-->>C: Mark Rider No-Show with timestamp C->>RD: Publish RiderNoShow event RD->>S: Fan-out to rider/admin S-->>R: No-show flag; driver can cancel S-->>A: No-show event for review C->>C: Calculate no-show fee based on policy end
Decline tracking: Each driver decline is logged and counted. After 3 declines, extended ETA warning is shown. After 5 declines, driver moved offline to prevent ride rejection abuse.
Reassignment logic: When a driver declines, Dispatch Engine immediately selects next eligible driver from pool based on distance, rating, and availability.
Rider visibility: Riders see transparency: if multiple drivers decline, they receive an alert that finding drivers is taking longer with an updated ETA.
No-Show timeout: Configurable wait threshold (default 5 min). Once exceeded, driver can mark Rider No-Show. No-show fee is calculated and captured for policy enforcement.
Event publishing: Decline count thresholds, driver offline, and no-show events all published to Redis for real-time admin monitoring.

9. Administrative Intervention & Force End Review

Admin review workflow for Force End trips, SOS events, and support escalations. Admins can review evidence, approve/reject decisions, and take corrective actions.

sequenceDiagram autonumber participant RA as Rider App participant DR as Driver App participant C as Core Service participant RD as Redis (pub/sub) participant S as SignalR Service participant A as Admin Portal participant DB as Postgres / Audit alt Force End Event DR->>C: Trip Force Ended (reason + proof) C->>RD: Publish ForceEndEvent (trip details, reason, location, proof) else SOS Escalation RA->>C: SOS triggered -> Admin escalates C->>RD: Publish SOSEscalation end RD->>S: Fan-out to admin subscribers S-->>A: Alert: Force End / SOS awaiting review A-->>A: Display: Trip timeline, location map, driver/rider notes, attached proof (photo/video) A->>C: Admin reviews and takes action alt Approve (Valid Exception) A->>C: Approve Force End action C->>DB: Log admin decision + timestamp C->>RD: Publish ForceEndApproved event RD->>S: Publish to rider and driver S-->>RA: Trip accepted as completed (no further action needed) S-->>DR: Force-end approved, no further action needed else Reject (Policy Violation Suspected) A->>C: Reject and flag driver for review C->>DB: Log rejection reason, escalate to compliance team C->>RD: Publish ForceEndRejected + DriverFlagged event RD->>S: Notify driver that trip is flagged S-->>DR: Driver receives notification of policy review else Request More Info A->>C: Request additional details from driver C->>S: Send chat/form request to driver S-->>DR: Admin requesting clarification on Force End DR->>C: Driver provides follow-up details C->>A: Updated details available in admin portal end A->>C: Admin can manually cancel/reassign for support cases C->>RD: Publish AdminIntervention event RD->>S: Update rider and driver of admin action S-->>RA: Trip cancel/reassigned by admin S-->>DR: Trip cancelled by admin C->>DB: Audit log: All admin actions, reason, timestamp, admin user ID
Real-time queue: Force End, SOS escalations, and disputed trips appear in Admin Portal in real-time via Redis pub/sub. No manual refresh needed.
Evidence handling: Admin can view attached proof (photos, videos, reason notes, trip location). Proof is stored securely and linked to trip record.
Three outcomes: Approve (trip completed legitimately), Reject (flag driver for policy review), Request More Info (gather additional evidence before deciding).
Compliance: All admin decisions are audit-logged with timestamp, admin user ID, and reason. Rejected trips escalate to compliance team for potential driver action.
Proactive support: Admin can manually cancel or reassign trips for support cases without waiting for driver exception. Action is immediately broadcast to both rider and driver.

10. Driver Registration End-to-End Flow

HEART FUNCTION

Driver onboarding is a core platform function: secure registration, KYC verification, admin approval, and activation gating before ride operations.

sequenceDiagram autonumber participant DR as Driver App participant C as Core Service participant S as Cloudflare Storage participant A as Admin Portal participant DB as PostgreSQL participant N as Notification Worker DR->>C: Submit mobile number for OTP C->>DR: OTP challenge DR->>C: Verify OTP alt OTP valid C->>DB: Create/continue onboarding draft profile DR->>C: Submit profile details and emergency contact DR->>C: Upload KYC docs metadata C->>S: Store KYC files securely C->>DB: Save KYC references + status UnderReview C->>A: Publish onboarding review task A->>C: Approve or Reject with reason alt Approved C->>DB: Set driver status Active C->>N: Send approval push/SMS N-->>DR: Account approved, can go Online else Rejected C->>DB: Set status Rejected with reason C->>N: Send rejection notice N-->>DR: Rejected, resubmit required fields end else OTP invalid C-->>DR: OTP failed with retry/cooldown policy end
Activation gate: Driver cannot switch to Online until admin approval sets account status to Active.
Auditability: Approval and rejection actions include actor, timestamp, and reason for compliance review.
Future AI-ready: AI validation can be inserted before admin approval as advisory scoring for document quality and fraud risk.

11. No-Loss Real-Time Reliability Controls

RESILIENCE

Real-time ride operations must survive disconnects and retries without losing critical state. The platform uses durable writes, idempotency, and replay-safe event delivery.

flowchart LR DR[Driver/Rider/Admin Clients] SR[SignalR Service] CS[Core Service] PG[(PostgreSQL)] RD[(Redis PubSub)] MON[Monitoring and Alerts] DR -->|state change request with idempotency key| CS CS -->|persist critical transition first| PG CS -->|publish event with event_id and sequence_no| RD RD -->|fan-out| SR SR -->|real-time delivery| DR DR -->|ack last sequence| SR SR -->|on reconnect request snapshot| CS CS -->|latest authoritative snapshot| DR CS --> MON SR --> MON RD --> MON
No-loss policy: Critical ride transitions are stored in PostgreSQL before real-time publish acknowledgement.
Idempotency: Start Trip, End Trip, Cancel, Force End, and No-Show APIs reject duplicate retries using idempotency keys.
Replay safety: Event payload includes trip_id, event_id, sequence_no, and timestamp to support ordering checks and dedupe.
Reconnect handling: On WebSocket break, clients reconnect and request latest snapshot so operations continue safely.
Operations guardrail: Alert on publish failures, consumer lag, reconnect spikes, and dead-letter queue growth.

12. Failure Scenarios Matrix

Operational reliability contract for core ride continuity and onboarding safety. Each scenario must have test evidence before production release.

Scenario Required Handling Target SLO Loss Tolerance
Mobile network drop during active trip Auto-reconnect + snapshot sync + sequence resume <= 10s normal recovery None for critical trip state
Redis lag or short outage Durable event record + publish retries + lag alerts <= 60s lag normalization None for critical trip state
Duplicate retries from app Idempotency key + dedupe by trip/action <= 2s duplicate rejection None
SignalR disconnect/restart Reconnect, resubscribe, replay snapshot <= 15s stream recovery None for critical trip state
Notification provider outage Retry with backoff + dead-letter + operator alert First retry <= 30s Only non-critical notifications
Driver onboarding duplicate submit Idempotent onboarding draft + document dedupe + audit trail No duplicate driver profile None
