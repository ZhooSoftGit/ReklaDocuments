End-to-End Technical Flow

1. Client sends request to Core Service hosted on Azure App Service.

2. Core Service applies business logic and persists data in PostgreSQL.

3. Core Service updates Redis for fast operational state.

4. For live updates, Core Service pushes events to SignalR Service, which broadcasts to clients.

5. For notifications, Core Service publishes events to Azure Event Bus.

6. Notification worker sends messages to OTP, WhatsApp, or push providers.

7. Files are stored in Cloudflare-managed storage and secrets are read from Azure Key Vault.

Heart Functions and Reliability Guardrails

Heart Function 1: Driver Registration End-to-End

  • OTP validation, profile capture, and mandatory KYC upload.
  • Admin approval/rejection with reason and audit trail.
  • Activation gating: no Online mode before approval.
  • Future-ready AI validation step before final admin activation.

Heart Function 2: Real-Time Ride with No Data Loss

  • Core Service is source of truth for critical state transitions.
  • Persist critical state before live publish acknowledgement.
  • Redis + SignalR fan-out for low-latency delivery to Rider, Driver, and Admin.
  • Reconnect, retry, idempotency, and dedupe controls for unstable networks.

Reliability controls: event_id + sequence_no on critical events, client reconnection with snapshot recovery, and retry-safe APIs using idempotency keys.

Failure posture: if network breaks, the trip state remains safe in PostgreSQL, then replay/resync restores client continuity.

Operations posture: monitor publish failures, lag, reconnect spikes, and dead-letter queues as mandatory production health signals.

Implementation Notes

  • Initial model: 2 services only (Core Service + SignalR Service).
  • Data ownership: Core Service is the source of truth for business logic and persistence.
  • Constraint: Azure Event Bus is used only for notification-related asynchronous handling.
  • Scalability: Core Service modules can be extracted later if scale requires.
